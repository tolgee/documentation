## Server settings \{#server-_-server}

Prefix: `server`

These properties are used to configure the server.

- ##### `port` \{#server-port-_-port}

  Port on which Tolgee exposes itself. (default: `8080`)

## Spring settings \{#spring-_-spring}

Prefix: `spring`

These properties are used to configure Spring framework.

### Data source settings \{#spring-datasource-_-datasource}

Prefix: `spring.datasource`

Since Tolgee is built on Spring framework, you have to edit following configuration props
to configure its database connection. These properties can be omitted when using
[Postgres autostart](#postgres-autostart), which is enabled by default.

- ##### `url` \{#spring-datasource-url-_-url}

  The url of the datasource in format `jdbc:postgresql://host:port/dbname`. e.g. `jdbc:postgresql://db:5432/postgres`

- ##### `username` \{#spring-datasource-username-_-username}

  Database username. e.g. `postgres`

- ##### `password` \{#spring-datasource-password-_-password}

  Database password. e.g. `postgres`

Prefix: `tolgee`

Configuration specific to Tolgee.

- ##### `file-storage-url` \{#tolgee-filestorageurl-_-filestorageurl}

  Public URL where Tolgee is accessible. Used to generate links to Tolgee (e.g. email confirmation link).

  Maximum amount of screenshots which can be uploaded per API key. (default: `20`)

  Maximum length of translations. (default: `10000`)

- ##### `max-upload-file-size` \{#tolgee-maxuploadfilesize-_-maxuploadfilesize}

### Authentication \{#tolgee-authentication-_-authentication}

Prefix: `tolgee.authentication`

Configuration of Tolgee's authentication.

- ##### `create-demo-for-initial-user` \{#tolgee-authentication-createdemoforinitialuser-_-createdemoforinitialuser}

  When enabled, creates Demo project and quick start guide for inital user
   (default: `true`)

- ##### `enabled` \{#tolgee-authentication-enabled-_-enabled}

  Whether authentication is enabled. When authentication is disabled, there are no login screen and no permission control. Users get automatically logged in as the administrator account of the Tolgee instance. This is very useful, when you want to use Tolgee on your local machine, or you just want to test it.  (default: `true` when running with Java directly, `false` when running via Docker.)

- ##### `initial-password` \{#tolgee-authentication-initialpassword-_-initialpassword}

  Password of initial user. If unspecified, a random password will be generated and stored in the `initial.pwd` file, located at the root of Tolgee's data path.
  
  :::tip
  Once set, you don't need to set this property anymore, and it is recommended you unset it so the password is not stored in plaintext in config files.
  
  If you set this field to something else, the initial user's password will be updated **unless the password was updated once via the Tolgee web UI**.
  :::
  
  

- ##### `initial-username` \{#tolgee-authentication-initialusername-_-initialusername}

  Username of initial user.
  
  :::tip
  Tolgee will ask for an email instead of a username - don't worry, just give it the username specified here.
  :::
  
   (default: `admin`)

- ##### `jwt-expiration` \{#tolgee-authentication-jwtexpiration-_-jwtexpiration}

  Expiration time of generated JWT tokens in milliseconds. (default: `604800000` = 7 days)

- ##### `jwt-secret` \{#tolgee-authentication-jwtsecret-_-jwtsecret}

  Secret used to sign JWT authentication tokens with. It will be generated automatically, if not provided. You will be fine with 64 characters long random string.
  
  :::warning
  When running multiple replicas, it is required to set it, or you will be constantly logged out otherwise.
  :::
  
  

- ##### `jwt-super-expiration` \{#tolgee-authentication-jwtsuperexpiration-_-jwtsuperexpiration}

  Expiration time of generated JWT tokens for superuser in milliseconds. (default: `3600000` = 1 hour)

- ##### `native-enabled` \{#tolgee-authentication-nativeenabled-_-nativeenabled}

  Whether to enable Tolgee-native authentication and registration.
  When set to `false`, users will only be able to register and login via third-party SSO options (e.g. OAuth). (default: `true`)

- ##### `needs-email-verification` \{#tolgee-authentication-needsemailverification-_-needsemailverification}

  Whether users need to verify their email addresses when creating their account. Requires a valid [SMTP configuration](#smtp). (default: `false`)

- ##### `registrations-allowed` \{#tolgee-authentication-registrationsallowed-_-registrationsallowed}

  Whether users are allowed to register on Tolgee.
  When set to `false`, existing users must send invites to projects to new users for them to be able to register.
  When SSO is enabled, users can still register via SSO, even if this setting is set to `false`. (default: `false`)

- ##### `secured-image-retrieval` \{#tolgee-authentication-securedimageretrieval-_-securedimageretrieval}

  Whether image assets should be protected by Tolgee. When enabled, all images are served with an access token valid for
  a set period of time to prevent unauthorized access to images. (default: `false`)

- ##### `secured-image-timestamp-max-age` \{#tolgee-authentication-securedimagetimestampmaxage-_-securedimagetimestampmaxage}

  Expiration time of a generated image access token in milliseconds. (default: `7200000` = 2 hours)

- ##### `user-can-create-organizations` \{#tolgee-authentication-usercancreateorganizations-_-usercancreateorganizations}

  Whether regular users are allowed to create organizations. When `false`, only administrators can create organizations.
  By default, when the user has no organization, one is created for them; this doesn't apply when this setting is set to `false`. In that case, the user without organization has no permissions on the server.
  
  When SSO authentication is enabled, users created by SSO don't have their own organization automatically created no matter the value of this setting. (default: `true`)

- ##### ~~`create-initial-user`~~ \{#tolgee-authentication-createinitialuser-_-createinitialuser}

  **Removed in:** 3.31.0

  If true, Tolgee creates initial user on first start-up. (default: `true`)

- ##### ~~`ldap.*`~~ \{#tolgee-authentication-ldap-_-ldap}

  **Removed in:** 3.31.0

  LDAP-related settings.

- ##### ~~`user-can-create-projects`~~ \{#tolgee-authentication-usercancreateprojects-_-usercancreateprojects}

  **Removed in:** 2.33.0

  Whether regular users are allowed to create projects. When disabled, only administrators can create projects (default: `true`)

#### Server wide Single Sign-On \{#tolgee-authentication-ssoglobal-_-ssoglobal}

Prefix: `tolgee.authentication.sso-global`

Single sign-on (SSO) is an authentication process that allows a user to access multiple applications with one set of login credentials. To use SSO in Tolgee, can either configure global SSO settings in this section or in refer to `sso-organizations` section for enabling the per Organization mode.

There is a significant difference between global and per organization SSO: Global SSO can handle authentication for all server users no matter which organizations they belong to, while per organization SSO can handle authentication only for users of the organization and such users cannot be members of any other organization. SSO users associated with per organization SSO have no rights to create or manage organizations. Global SSO users should be invited to organizations they need to have access to. Per organization SSO users are automatically added to the organization they belong to.

- ##### `authorization-uri` \{#tolgee-authentication-ssoglobal-authorizationuri-_-authorizationuri}

  URL to redirect users for authentication

- ##### `client-id` \{#tolgee-authentication-ssoglobal-clientid-_-clientid}

  Unique identifier for an application

- ##### `client-secret` \{#tolgee-authentication-ssoglobal-clientsecret-_-clientsecret}

  Key used to authenticate the application

- ##### `custom-login-text` \{#tolgee-authentication-ssoglobal-customlogintext-_-customlogintext}

  Custom text for the SSO login page.

- ##### `custom-logo-url` \{#tolgee-authentication-ssoglobal-customlogourl-_-customlogourl}

  Custom logo URL to be displayed on the login screen. Can be set only when `nativeEnabled` is `false`.

- ##### `domain` \{#tolgee-authentication-ssoglobal-domain-_-domain}

  Used to identify the organization on login page

- ##### `enabled` \{#tolgee-authentication-ssoglobal-enabled-_-enabled}

  Enables SSO authentication on global level - as a login method for the whole server (default: `false`)

- ##### `force` \{#tolgee-authentication-ssoglobal-force-_-force}

  When true, users with an email matching the organization's domain must sign in using SSO (default: `false`)

- ##### `session-expiration-minutes` \{#tolgee-authentication-ssoglobal-sessionexpirationminutes-_-sessionexpirationminutes}

  Minutes after which the server will recheck the user's with the SSO provider to ensure the user account is still valid. This is to prevent the user from being able to access the server after the account has been disabled or deleted in the SSO provider. (default: `10`)

- ##### `token-uri` \{#tolgee-authentication-ssoglobal-tokenuri-_-tokenuri}

  URL for exchanging authorization code for tokens

#### Per-Organization Single Sign-On \{#tolgee-authentication-ssoorganizations-_-ssoorganizations}

Prefix: `tolgee.authentication.sso-organizations`

Single sign-on (SSO) is an authentication process that allows a user to access multiple applications with one set of login credentials. To use SSO in Tolgee, can either configure global SSO settings in `sso-global` section or in the per Organization mode by setting the `enable` to `true` in this section and configuring it separately for each organization in the organization settings.

There is a significant difference between global and per organization SSO: Global SSO can handle authentication for all server users no matter which organizations they belong to, while per organization SSO can handle authentication only for users of the organization and such users cannot be members of any other organization. SSO users associated with per organization SSO have no rights to create or manage organizations. Global SSO users should be invited to organizations they need to have access to. Per organization SSO users are automatically added to the organization they belong to.

- ##### `enabled` \{#tolgee-authentication-ssoorganizations-enabled-_-enabled}

  Enables SSO authentication (default: `false`)

- ##### `session-expiration-minutes` \{#tolgee-authentication-ssoorganizations-sessionexpirationminutes-_-sessionexpirationminutes}

  Minutes after which the server will recheck the user's with the SSO provider to ensure the user account is still valid. This is to prevent the user from being able to access the server after the account has been disabled or deleted in the SSO provider. (default: `10`)

### Cache \{#tolgee-cache-_-cache}

Prefix: `tolgee.cache`

At the expense of higher memory footprint, Tolgee can use a cache to reduce the stress on the database and fetch the
data it needs faster. Cache is also used to track certain states, such as rate limits.

- ##### `caffeine-max-size` \{#tolgee-cache-caffeinemaxsize-_-caffeinemaxsize}

  Maximum size of the Caffeine cache. When exceeded, some entries will be purged from cache. Set to -1 to disable size limitation.
  This has no effect when Redis cache is used. See
  [Caffeine's documentation about size-based eviction](https://github.com/ben-manes/caffeine/wiki/Eviction#size-based) (default: `-1`)

- ##### `clean-on-startup` \{#tolgee-cache-cleanonstartup-_-cleanonstartup}

  Whether to clean the cache on Tolgee startup (default: `true`)

- ##### `default-ttl` \{#tolgee-cache-defaultttl-_-defaultttl}

  TTL of cache data, in milliseconds. (default: `7200000` â‰ˆ 2 hours)

- ##### `enabled` \{#tolgee-cache-enabled-_-enabled}

  Whether Tolgee should use a cache. (default: `false`)

- ##### `use-redis` \{#tolgee-cache-useredis-_-useredis}

  Whether Tolgee should use Redis to store cache data instead of storing it in-memory.
  :::info
  In a distributed environment, you **should** use a Redis server to ensure consistent enforcement of rate limits, as
  they heavily rely on cache. For a simple single-node deployment, in-memory cache is sufficient.
  :::
  
  :::info
  Tolgee uses [Redisson](https://github.com/redisson/redisson) to interface with the Redis server. You can find the properties Redisson expects [here](https://github.com/redisson/redisson/tree/56ea7f5/redisson-spring-boot-starter#2-add-settings-into-applicationsettings-file).
  :::
  
   (default: `false`)

#### providers \{#tolgee-llm-providers-_-providers}


    List of LLM providers. Example:
    
    ``` yaml
    providers:
      - name: openai-gpt-4o-mini
        type: OPENAI
        api-key: "myApiKey"
        api-url: "https://api.openai.com"
        model: gpt-4o-mini
        format: "json_schema"
    ```
    
    or using environment variables:
    
    ```
    TOLGEE_LLM_PROVIDERS_0_NAME=MySuperDuperAI
    TOLGEE_LLM_PROVIDERS_0_TYPE=OPENAI
    TOLGEE_LLM_PROVIDERS_0_API_KEY=myApiKey
    ```

### Rate limits \{#tolgee-ratelimit-_-ratelimit}

Prefix: `tolgee.rate-limits`

Configuration of the rate limit behavior of the server.
Rate limits are used to protect against server overload and/or abuse, and are enabled by default.

You can disable global, per-endpoint and auth-related rate limits, or configure global rate limits.
Per-endpoint and auth-related rate limits are fixed and cannot be configured.



- ##### `authentication-limits` \{#tolgee-ratelimit-authenticationlimits-_-authenticationlimits}

  Control whether per-endpoint limits on authentication-related endpoints are enabled or not.
  :::warning
  It is **strongly** recommended to keep these limits enabled. They act as a protection layer against
  brute-force attacks on the login (and register) prompt.
  :::
  
   (default: `true`)

- ##### `email-verification-request-limit` \{#tolgee-ratelimit-emailverificationrequestlimit-_-emailverificationrequestlimit}

- ##### `email-verification-request-limit-enabled` \{#tolgee-ratelimit-emailverificationrequestlimitenabled-_-emailverificationrequestlimitenabled}

- ##### `email-verification-request-window` \{#tolgee-ratelimit-emailverificationrequestwindow-_-emailverificationrequestwindow}

- ##### `enabled` \{#tolgee-ratelimit-enabled-_-enabled}

  To turn all rate limits off, set this value to `false`.
  
  :::danger
  This field is **deprecated** and will be removed with Tolgee 4. If set to `false`, it will take priority
  over the more granular `global-limits`, `endpoint-limits` and `authentication-limits` and all limits
  will be disabled.
  :::
  
   (default: `true`)

- ##### `endpoint-limits` \{#tolgee-ratelimit-endpointlimits-_-endpointlimits}

  Control whether per-endpoint limits on the API are enabled or not.
  Does not affect authentication-related endpoints, these are controlled by `authentication-limits`. (default: `true`)

- ##### `global-limits` \{#tolgee-ratelimit-globallimits-_-globallimits}

  Control whether global limits on the API are enabled or not. (default: `true`)

- ##### `ip-request-limit` \{#tolgee-ratelimit-iprequestlimit-_-iprequestlimit}

  Amount of requests an IP address can do in a single time window. (default: `20000`)

- ##### `ip-request-window` \{#tolgee-ratelimit-iprequestwindow-_-iprequestwindow}

  Size, in milliseconds, of the time window for IP-based limiting. (default: `300000` = 5 minutes)

- ##### `user-request-limit` \{#tolgee-ratelimit-userrequestlimit-_-userrequestlimit}

  Amount of requests a user can do in a single time window. (default: `400`)

- ##### `user-request-window` \{#tolgee-ratelimit-userrequestwindow-_-userrequestwindow}

  Size, in milliseconds, of the time window for user-based limiting. (default: `60000` = 1 minute)

### Authentication \{#tolgee-authentication-_-authentication}

  When enabled, creates Demo project and quick start guide for inital user
   (default: `true`)

- ##### `ENABLED` \{#tolgee-authentication-enabled-_-enabled}

  Whether authentication is enabled. When authentication is disabled, there are no login screen and no permission control. Users get automatically logged in as the administrator account of the Tolgee instance. This is very useful, when you want to use Tolgee on your local machine, or you just want to test it.  (default: `true` when running with Java directly, `false` when running via Docker.)

- ##### `INITIAL_PASSWORD` \{#tolgee-authentication-initialpassword-_-initialpassword}

  :::tip

  If you set this field to something else, the initial user's password will be updated **unless the password was updated once via the Tolgee web UI**.
  :::

  Username of initial user.
  
  :::tip
  Tolgee will ask for an email instead of a username - don't worry, just give it the username specified here.
  :::
  
   (default: `admin`)

  Secret used to sign JWT authentication tokens with. It will be generated automatically, if not provided. You will be fine with 64 characters long random string.
  
  :::warning
  When running multiple replicas, it is required to set it, or you will be constantly logged out otherwise.

### Rate limits \{#tolgee-ratelimit-_-ratelimit}

Prefix: `TOLGEE_RATE_LIMITS`

Configuration of the rate limit behavior of the server.
Rate limits are used to protect against server overload and/or abuse, and are enabled by default.

You can disable global, per-endpoint and auth-related rate limits, or configure global rate limits.
Per-endpoint and auth-related rate limits are fixed and cannot be configured.



- ##### `AUTHENTICATION_LIMITS` \{#tolgee-ratelimit-authenticationlimits-_-authenticationlimits}

  Control whether per-endpoint limits on authentication-related endpoints are enabled or not.
  :::warning
  It is **strongly** recommended to keep these limits enabled. They act as a protection layer against
  brute-force attacks on the login (and register) prompt.
  :::
  
   (default: `true`)

- ##### `EMAIL_VERIFICATION_REQUEST_LIMIT` \{#tolgee-ratelimit-emailverificationrequestlimit-_-emailverificationrequestlimit}

- ##### `EMAIL_VERIFICATION_REQUEST_LIMIT_ENABLED` \{#tolgee-ratelimit-emailverificationrequestlimitenabled-_-emailverificationrequestlimitenabled}

- ##### `EMAIL_VERIFICATION_REQUEST_WINDOW` \{#tolgee-ratelimit-emailverificationrequestwindow-_-emailverificationrequestwindow}

- ##### `ENABLED` \{#tolgee-ratelimit-enabled-_-enabled}

  To turn all rate limits off, set this value to `false`.
  
  :::danger
  This field is **deprecated** and will be removed with Tolgee 4. If set to `false`, it will take priority
  over the more granular `global-limits`, `endpoint-limits` and `authentication-limits` and all limits
  will be disabled.
  :::
  
   (default: `true`)

- ##### `ENDPOINT_LIMITS` \{#tolgee-ratelimit-endpointlimits-_-endpointlimits}

  Control whether per-endpoint limits on the API are enabled or not.
  Does not affect authentication-related endpoints, these are controlled by `authentication-limits`. (default: `true`)

- ##### `GLOBAL_LIMITS` \{#tolgee-ratelimit-globallimits-_-globallimits}

  Control whether global limits on the API are enabled or not. (default: `true`)

- ##### `IP_REQUEST_LIMIT` \{#tolgee-ratelimit-iprequestlimit-_-iprequestlimit}

  Amount of requests an IP address can do in a single time window. (default: `20000`)

- ##### `IP_REQUEST_WINDOW` \{#tolgee-ratelimit-iprequestwindow-_-iprequestwindow}

  Size, in milliseconds, of the time window for IP-based limiting. (default: `300000` = 5 minutes)

- ##### `USER_REQUEST_LIMIT` \{#tolgee-ratelimit-userrequestlimit-_-userrequestlimit}

  Amount of requests a user can do in a single time window. (default: `400`)

- ##### `USER_REQUEST_WINDOW` \{#tolgee-ratelimit-userrequestwindow-_-userrequestwindow}

  Size, in milliseconds, of the time window for user-based limiting. (default: `60000` = 1 minute)

### Postgres autostart \{#tolgee-postgresautostart-_-postgresautostart}

Prefix: `tolgee.postgres-autostart`

Defines whether and how is PostgreSQL started on Tolgee startup.

- ##### `container-name` \{#tolgee-postgresautostart-containername-_-containername}

  The container name of the Postgres container.
  This setting is applicable only for `DOCKER` mode. (default: `tolgee_postgres`)

- ##### `database-name` \{#tolgee-postgresautostart-databasename-_-databasename}

  The name of the database created to store Tolgee data. (default: `postgres`)

- ##### `enabled` \{#tolgee-postgresautostart-enabled-_-enabled}

  Whether to start PostgreSQL on Tolgee startup. (default: `true`)

- ##### `max-wait-time` \{#tolgee-postgresautostart-maxwaittime-_-maxwaittime}

  The max time to wait for running postgres in seconds. (default: `300`)

- ##### `mode` \{#tolgee-postgresautostart-mode-_-mode}

  How is Tolgee running PostgreSQL.
  
  Options:
  - `DOCKER` - Tolgee tries to run Postgres Docker container in your machine. This is default option when running Tolgee using Java. See [Running with Java](/self_hosting/running_with_java.mdx).
  - `EMBEDDED` - Tolgee tries to run it's embedded PostgreSQL which is bundled in the `tolgee/tolgee` Docker image. (default: `DOCKER`)

- ##### `password` \{#tolgee-postgresautostart-password-_-password}

  Database password to bootstrap Postgres with. (default: `postgres`)

- ##### `port` \{#tolgee-postgresautostart-port-_-port}

  The port of Postgres to listen on host machine.
  This setting is applicable only for `DOCKER` mode. (default: `25432`)

- ##### `stop` \{#tolgee-postgresautostart-stop-_-stop}

  When true, Tolgee will stop the Postgres container on Tolgee shutdown. This setting is applicable only for `DOCKER` mode. (default: `true`)

- ##### `user` \{#tolgee-postgresautostart-user-_-user}

  Database user to bootstrap Postgres with. (default: `postgres`)

### Rate limits \{#tolgee-ratelimit-_-ratelimit}

Prefix: `tolgee.rate-limits`

Configuration of the rate limit behavior of the server.
Rate limits are used to protect against server overload and/or abuse, and are enabled by default.

You can disable global, per-endpoint and auth-related rate limits, or configure global rate limits.
Per-endpoint and auth-related rate limits are fixed and cannot be configured.



- ##### `authentication-limits` \{#tolgee-ratelimit-authenticationlimits-_-authenticationlimits}

  Control whether per-endpoint limits on authentication-related endpoints are enabled or not.
  :::warning
  It is **strongly** recommended to keep these limits enabled. They act as a protection layer against
  brute-force attacks on the login (and register) prompt.
  :::
  
   (default: `true`)

- ##### `email-verification-request-limit` \{#tolgee-ratelimit-emailverificationrequestlimit-_-emailverificationrequestlimit}

- ##### `email-verification-request-limit-enabled` \{#tolgee-ratelimit-emailverificationrequestlimitenabled-_-emailverificationrequestlimitenabled}

- ##### `email-verification-request-window` \{#tolgee-ratelimit-emailverificationrequestwindow-_-emailverificationrequestwindow}

- ##### `enabled` \{#tolgee-ratelimit-enabled-_-enabled}

  To turn all rate limits off, set this value to `false`.
  
  :::danger
  This field is **deprecated** and will be removed with Tolgee 4. If set to `false`, it will take priority
  over the more granular `global-limits`, `endpoint-limits` and `authentication-limits` and all limits
  will be disabled.
  :::
  
   (default: `true`)

- ##### `endpoint-limits` \{#tolgee-ratelimit-endpointlimits-_-endpointlimits}

  Control whether per-endpoint limits on the API are enabled or not.
  Does not affect authentication-related endpoints, these are controlled by `authentication-limits`. (default: `true`)

- ##### `global-limits` \{#tolgee-ratelimit-globallimits-_-globallimits}

  Control whether global limits on the API are enabled or not. (default: `true`)

- ##### `ip-request-limit` \{#tolgee-ratelimit-iprequestlimit-_-iprequestlimit}

  Amount of requests an IP address can do in a single time window. (default: `20000`)

- ##### `ip-request-window` \{#tolgee-ratelimit-iprequestwindow-_-iprequestwindow}

  Size, in milliseconds, of the time window for IP-based limiting. (default: `300000` = 5 minutes)

- ##### `user-request-limit` \{#tolgee-ratelimit-userrequestlimit-_-userrequestlimit}

  Amount of requests a user can do in a single time window. (default: `400`)

- ##### `user-request-window` \{#tolgee-ratelimit-userrequestwindow-_-userrequestwindow}

  Size, in milliseconds, of the time window for user-based limiting. (default: `60000` = 1 minute)
