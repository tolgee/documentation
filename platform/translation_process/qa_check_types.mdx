This page describes each QA check type in detail, explaining what problems it detects and when it is useful. To learn how to enable and configure these checks, see [QA Checks](./qa_checks.mdx).

## Comparison Checks

Comparison checks compare each translation against the [base language](/platform/projects_and_organizations/languages) translation. They help ensure that structural elements like placeholders, formatting, and punctuation are consistent across all languages.

### Empty Translation

Detects translations that are blank or contain only whitespace. This check is **off by default** because empty translations are sometimes intentional (e.g., when a key doesn't need translation in a particular language). Enable it if you want to ensure every key has a translation in every language.

### Spaces Mismatch

Detects differences in spacing between the base translation and the target. This includes leading or trailing spaces that were added or removed, doubled spaces, and non-breaking space inconsistencies. Spacing issues can cause layout problems in the UI, especially in languages where text is concatenated with other elements.

### Unmatched New Lines

Detects missing or extra line breaks compared to the base translation. Newlines often matter for formatting in multi-line UI elements, tooltips, or email templates. A missing or extra newline can break the intended layout.

### Character Case Mismatch

Detects capitalization differences between the base and target translation, such as a lowercase first letter when the base starts with uppercase. This is particularly useful for UI labels and headings where consistent capitalization is expected.

### Missing Numbers

Detects numeric values that are present in the base translation but missing in the target. Numbers often represent important data like counts, prices, or version numbers. A missing number usually means the translator accidentally deleted it.

### Punctuation Mismatch

Detects missing, extra, or replaced punctuation marks compared to the base translation. Consistent punctuation is important for a polished user experience. For example, if the base translation ends with a period, the target should generally end with one too.

### Brackets Mismatch

Detects brackets (parentheses, square brackets, curly braces) that are missing or extra compared to the base translation. Brackets often contain important contextual information or are part of technical syntax.

### Brackets Unbalanced

Detects unclosed or unopened brackets within the translation itself, regardless of the base translation. An unclosed bracket is almost always a mistake and can cause display issues or parsing errors.

### Special Character Mismatch

Detects special characters that are missing or extra compared to the base translation. This check is **off by default** because special character usage can legitimately differ between languages. Enable it when your translations contain technical symbols or characters that must be preserved exactly.

### Different URLs

Detects URLs that are missing, extra, or different from those in the base translation. URLs in translations typically link to resources that should remain consistent across languages. A changed or missing URL can lead to broken links.

### Key Length Limit

Detects translations that exceed the [character limit](/platform/translation_keys/keys#character-limit) configured on the translation key. Character limits are often set to match UI constraints (e.g., button widths, text fields). Exceeding the limit can cause text to overflow or get truncated.

### Spelling

Detects spelling errors using LanguageTool. Spelling mistakes make translations look unprofessional and can confuse users. This check supports many languages and adapts to the target language automatically.

### Grammar

Detects grammar errors using LanguageTool. Grammar issues can change the meaning of a sentence or make it harder to understand. Like the spelling check, this adapts to the target language.

### Repeated Words

Detects consecutive repeated words (e.g., "the the" or "is is"). Repeated words are almost always accidental and are a common copy-paste or editing mistake.

## Syntax Checks

Syntax checks validate the structure of the translation itself. They catch issues that would cause runtime errors or broken output in the application.

### Inconsistent Placeholders

Detects variable placeholders (e.g., `{userName}`, `{count}`) that are missing or extra compared to the base translation. Missing placeholders cause runtime errors or display raw placeholder syntax to users. Extra placeholders that don't exist in the code will also show as raw text. See [Tolgee Universal ICU Placeholders](/platform/translation_process/tolgee_universal_icu_placeholders) for more about how Tolgee handles placeholders.

### Inconsistent HTML

Detects HTML tags that are missing or extra compared to the base translation. If your translations contain HTML markup for formatting (bold, links, etc.), missing tags can break the intended formatting or even the page layout.

### HTML Syntax

Detects unclosed or unopened HTML tags within the translation. For example, a `<b>` without a matching `</b>`. Broken HTML can cause rendering issues that cascade through the rest of the page.

### ICU Syntax

Validates [ICU message format](/platform/translation_process/icu_message_format) syntax, including `select`, `plural`, and `choice` expressions. ICU messages have strict syntax requirements, and even a small mistake (like a missing closing brace) will cause the entire message to fail at runtime, potentially crashing the application or showing an error to users.

### Unresolved Comments

Flags translations that have unresolved [comments](/platform/translation_process/comments). This check ensures that feedback from reviewers or other translators is addressed before a translation is considered complete. It is especially useful in team workflows where comments are used for discussion and review.
